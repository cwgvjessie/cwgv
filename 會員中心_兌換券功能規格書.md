# 會員中心 - 兌換碼功能規格書 (Redemption Code Functional Specification)

## 1. 專案背景與目標
為了統合「遠見」會員權益之發放與核銷流程，本次開發旨在會員中心新增兩套獨立的兌換機制。透過統一的後台產生代碼後，依據代碼屬性能分別對應「數位商品發放」或「現有訂閱延長」兩大主要場景。

---

## 2. 兩大功能類別摘要 (Overview)

### 2.1 【功能一】折價券夾 (Discount Voucher Center)
主要用於發放一般的數位商品或活動優惠，使用者需將代碼「歸戶」為票券後使用。
*   **介面預覽**: [折價券頁面](https://cwgvjessie.github.io/cwgv/voucher_wireframe.html)
*   **標準用語**: **「折價券」** (票券)、**「折價碼」** (輸入代碼)。

### 2.2 【功能二】權益兌換 (Benefit Redemption)
專供會員透過代碼直接兌換相對應的「帳號權益」，不產生實體票券。
*   **介面預覽**: [訂戶管理頁面](https://cwgvjessie.github.io/cwgv/subscription_management.html)
*   **標準用語**: **「兌換券」** (通用)、**「兌換碼」** (輸入代碼)。
*   **適用權益範圍**: 包含但不限於「訂閱期限延長」、「特定專題/數位內容閱讀權」、「實體活動報名資格」等。

### 2.3 兌換資格限制 (Redemption Eligibility)
不論功能一或功能二，皆遵循以下檢核規則：
1.  **基本規則**: **單一活動/批次每人限領一次**。
2.  **進階限制**: 可於後台額外設定「**限未曾兌換過之新用戶**」使用。系統將檢核該帳號過去是否曾有過任何本系統之兌換紀錄。

---

## 3. 【功能一】折價券夾詳細規格

### 3.1 使用者介面架構
*   **標題**: 顯示「折價券」標題，右側顯示「New」標籤。
*   **輸入框**: Placeholder 為「輸入折價碼」。
*   **提示文字**:
    ```
    *點擊「新增」按鈕，或是按enter鍵，即可將折價券加入會員帳戶
    *如新增後未在「可使用」頁籤中看到折價券，請嘗試重新整理或退出頁面後重新進入
    ```

### 3.2 列表與籤頁 (Tabs)
包含四個狀態籤頁，系統需依據「啟用日期」與「使用狀態」自動分類：
1.  **可使用 (Available)**: 已歸戶且在有效期內，尚未使用的票券。顯示「立即使用」按鈕。
2.  **待啟用 (Future)**: 已歸戶但尚未到達開始時間。顯示「YYYY/MM/DD啟用」字樣。
3.  **已失效 (Expired)**: 已歸戶但超過使用期限。顯示「已失效」。
4.  **已使用 (Used)**: 已經使用過的票券。顯示「已使用」。

### 3.3 功能詳細邏輯
*   **新增邏輯 (Claiming)**:
    1.  支援 URL 參數 `?code=VALID_CODE` 自動填入。
    2.  呼叫後端 API 檢核代碼有效性（期限、餘額、資格）。
    3.  **成功**: 顯示 Toast「✅ 兌換成功！已歸戶至您的帳戶」，清空輸入框，自動切換至「可使用」分頁並置頂新券。
    4.  **失敗**: 顯示 Toast「❌ 兌換失敗：[原因]」(包含：過期、查無代碼、已被使用、未輸入)。

### 3.4 票券卡片 (Ticket Card)
*   **票券設計**: 具備票券缺口 (Ticket Notch) 設計。
*   **主要定義**:
    - **標題**: 商品名稱。
    - **代碼內容**: 加粗顯示。
    - **有效期限**: 例如 "即日起 ~ 無使用期限" 或 "2026/01/01 ~ 2026/12/31"。
*   **操作**: 「立即使用」按鈕。點擊後彈出確認視窗，確認後顯示導向 Toast 並於 1 秒後導向商品連結。

## 4. 【功能二】權益兌換詳細規格

### 4.1 兌換邏輯
*   **用語**: 統一使用「兌換碼/券」。
*   **核銷效果**: 成功後不歸戶票券，而是直接於資料庫中變更或新增該會員之相關權益紀錄。
*   **核心應用情境 (範例)**:
    - **訂閱延長**: 將該會員之「訂閱到期日」與「下次扣款日」往後順延。
    - **內容開通**: 開通特定數位產品之存取權限。

### 4.2 UI 反饋與狀態 (Redemption Feedback)
兌換時需檢核並透過 Toast 顯示對應狀態：
*   **成功 (Success)**: 顯示「✅ 兌換成功！已成功取得 [對應權益名稱]」。
    - 若為訂閱延長，到期日期需立即變更高亮。
    - 若為其他權益，應引導使用者至對應頁面查看。
*   **過期 (Expired)**: 顯示「❌ 兌換失敗：此兌換碼已過期」。
*   **查無代碼 (Invalid)**: 顯示「❌ 兌換失敗：查無此兌換碼，請確認輸入」。
*   **已被使用 (Used)**: 顯示「❌ 兌換失敗：此兌換碼已被使用」。
*   **新用戶限制 (Constraint)**: 若該活動限新戶且會員不符資格，顯示「❌ 兌換失敗：此活動僅限新用戶領取」。

## 5. 後台管理系統 - 兌換券設定 (Redemption Settings)

管理者僅需進行發行批次的配置：

### 5.1 批次配置內容
*   **批次名稱**: 活動識別名稱。
*   **序號類型**:
    - **固定序號 (Fixed)**: 所有人共用一碼，直到組數上限。
    - **隨機序號 (Random)**: 批次產生不重複亂數。
*   **自定義前綴**: 產生隨機碼時可設定開頭文字 (如 `GVM-`)。
*   **對應權益內容**: 定義此批次代碼兌換後的具體權益內容。
*   **兌換資格限制**: 
    - 預設勾選「每人限領一次」。
    - 可加選「僅限從未兌換過任何權益的新用戶使用」。
*   **兌換組數上限**: 本批次代碼的最大兌換次數。
*   **有效期限**: 代碼可被輸入兌換的最後日期。

## 6. 技術規範與安全性 (System & Tech)

### 6.1 資料表建議
*   `Vouchers`: 定義批次與代碼 (ID, Code, Type, Limit, Start/EndTime, TargetURL)。
*   `User_Vouchers`: 持有與核銷紀錄 (ID, UserID, VoucherID, Status, ClaimedAt, UsedAt)。

### 6.2 安全性與資格限制邏輯
*   **代碼唯一性 (Unique Index)**: 所有產生的代碼在資料庫中必須為唯一。
*   **核心檢核機制**:
    1.  **批次狀態**: 檢核活動是否在有效期內且未停用。
    2.  **領取上限**: 檢核本批次已領取總數是否達到上限。
    3.  **身份重複性**: 檢核同一會員資帳號是否曾於「本批次活動」中領過。
    4.  **新老客戶過濾**: 若開啟限制，系統需比對 `User_Vouchers` 資料表，確認該會員帳號在「全系統」中是否曾有過領取紀錄。
*   **防暴力破解**: 需限制單一帳號/IP 連續輸入錯誤代碼之頻率，超過次數需暫時鎖定或加入驗證碼機制。
